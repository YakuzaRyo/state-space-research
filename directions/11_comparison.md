## 研究历程

### 2026-03-10 11:46 对比分析深度研究（第二轮）

#### 深入分析：现有AI编码工具的架构范式

**现有主流工具的共同范式：LLM-as-Agent**
- Claude Code、Cursor、Windsurf 等都采用 "LLM + 工具集 + Prompt引导" 模式
- LLM 是核心决策者，工具是执行手脚，Prompt 是行为约束
- 这种范式的根本问题：**LLM 的"自由意志"与工具的安全边界之间的张力**

#### 根本缺陷的更深层分析

**缺陷1：软约束的根本脆弱性**
```
用户意图 → [Prompt/CLAUDE.md] → LLM理解 → 行动选择
                              ↓
                       LLM可能"理解错"
                              ↓
                       执行错误操作
```
- 问题：Prompt 是"建议"，不是"规则"
- LLM 可以"合理地"忽略不明确的约束
- 即使明确的约束，LLM 也可能在特定上下文中"灵活处理"

**缺陷2：验证的滞后性**
```
生成代码 → 运行测试 → 发现错误 → 重新生成 → 再次验证
    ↓           ↓
  事后验证    可能多次循环
```
- 测试驱动开发 (TDD) 是补丁，不是架构解决方案
- 状态空间架构在生成前就约束输出空间

**缺陷3：工具能力的失控风险**
- Claude Code 的 `Bash` 工具可以执行任意 shell 命令
- MCP 协议扩展了能力边界，但也扩展了攻击面
- 权限系统是动态补丁，但无法穷举所有危险操作

**缺陷4：状态的黑盒特性**
- LLM 的内部状态不可观测、不可审计
- "幻觉"问题无法从架构上解决
- 决策过程不可解释、不可回滚

#### 状态空间架构的范式优势

```
用户意图 → [类型约束 + API边界] → 状态空间导航 → 有效输出
                      ↓
              硬性边界（物理约束）
                      ↓
              无效状态无法构造
```

**关键区别：**
| 维度 | 现有架构 (LLM-as-Agent) | 状态空间架构 |
|------|------------------------|--------------|
| **约束来源** | Prompt（建议性） | 类型系统 + API边界（强制性） |
| **验证时机** | 生成后测试 | 生成前约束 |
| **状态管理** | LLM 内部黑盒 | 显式状态空间 |
| **错误处理** | 运行时发现 | 编译期/入口拒绝 |
| **灵活性** | 高（但不可控） | 受限但安全 |

#### OpenCode 架构分析（推测）

基于行业观察，OpenCode 类的工具通常：
- 采用更激进的自动化策略
- 依赖更强大的上下文理解
- 尝试减少用户干预
- 但仍面临相同的架构问题

**可能的改进方向：**
1. 结构化输出约束（类似 XGrammar）
2. 形式化验证集成（类似 Refine4LLM）
3. 确定性执行环境（类似 Praetorian）

#### 状态空间架构的应对策略

**硬性边界实现机制：**

1. **类型安全的编译期保证**
   - Rust/OCaml 类型系统排除无效状态
   - LLM 只能构造类型系统允许的值
   - 编译期发现错误，而非运行时

2. **API 边界的物理约束**
   - 危险操作不暴露给 LLM
   - 沙盒化执行环境
   - 确定性 Hooks 在 LLM 上下文外强制执行

3. **状态转移的可验证性**
   - 每步操作在预定义状态空间内
   - 状态转移规则明确、可审计
   - 失败快速：无效操作入口拒绝

4. **组合式安全**
   - 小型、可验证的组件组合
   - 不依赖 LLM 的"理解"或"遵守"
   - 系统级保证，而非个体级约束

#### 关键洞察：从"信任LLM"到"信任系统"

**现有架构的隐含假设：**
> "如果我们给 LLM 足够好的 Prompt/约束，它就会正确行事"

**状态空间架构的前提：**
> "无论 LLM 如何理解，它都物理上无法违反约束"

这是一个根本性的范式转变：
- ❌ 软约束："请你不要这样做"（LLM 可能不听）
- ✅ 硬边界："你不能这样做"（LLM 物理上做不到）

#### 潜在挑战与权衡

**状态空间架构的挑战：**

1. **灵活性降低**
   - 硬性边界限制了 LLM 的创造性
   - 某些"意外"的好结果可能无法出现
   - 需要精心设计状态空间

2. **类型系统复杂度**
   - 状态空间设计需要专业知识
   - 覆盖所有场景很困难
   - 类型错误可能难以调试

3. **与现有工具链集成**
   - 需要新工具，而非简单复用
   - 学习曲线陡峭
   - 生态系统尚未成熟

4. **性能开销**
   - 编译期检查有额外开销
   - 类型系统运行时可能有开销

**何时选择状态空间架构：**
- 安全关键系统
- 需要确定性行为的场景
- 错误代价高于灵活性损失的场景

**何时选择现有架构：**
- 探索性开发
- 快速原型
- 灵活性比安全性更重要的场景

### 2026-03-10 对比分析深度研究

#### 核心发现：Claude Code架构分析

**Claude Code 的关键特性：**
1. **Agentic Coding Tool** - 读取代码库、编辑文件、运行命令
2. **多端支持** - 终端CLI、VS Code、桌面应用、浏览器
3. **MCP (Model Context Protocol)** - 连接外部数据源的开放标准
4. **CLAUDE.md** - 项目级指令文件

**Claude Code的工作模式分析：**
- 基于自然语言Prompt驱动
- 权限请求系统（Permission Request）用于控制危险操作
- 工具调用（Tool Use）通过 `Read/Search/Edit/Bash` 等原子操作
- 测试验证驱动开发

#### 与状态空间架构的根本差异

| 维度 | Claude Code | 状态空间架构 |
|------|------------|-------------|
| **约束机制** | 自然语言Prompt + 权限请求 | 编译期类型安全 + API边界 |
| **验证时机** | 事后验证（生成后检查） | 事前验证（生成空间内保证） |
| **状态管理** | 黑盒，依赖LLM内部状态 | 显式状态空间，状态转移可追踪 |
| **错误处理** | LLM理解指令，可能产生错误 | 无效操作在入口被拒绝 |
| **可扩展性** | 依赖Prompt工程 | 依赖类型系统扩展 |

#### 根本缺陷分析

**1. Prompt依赖的脆弱性**
- Claude Code依赖 `CLAUDE.md` 等自然语言文件来指导行为
- 这些约束本质上是"软约束"——LLM可能不完全遵守
- 没有物理上的强制力

**2. 事后验证的低效性**
- 代码生成后通过测试来验证正确性
- 如果测试失败，需要重新生成
- 没有在生成过程中约束输出的结构

**3. 状态不透明**
- 难以精确追踪LLM的决策过程
- 调试和回滚困难
- "幻觉"问题无法从架构上解决

**4. 工具能力的双刃剑**
- 强大的工具能力（如 `Bash` 命令）带来风险
- 权限系统是补丁，不是架构级别的解决方案

#### 状态空间架构的应对策略

**硬性边界实现：**
1. **类型安全** - 编译期排除无效状态，LLM无法构造类型系统不允许的值
2. **API边界约束** - LLM只能调用受限API，物理上无法访问危险操作
3. **确定性Hooks** - 在LLM上下文外强制执行（如Praetorian的Gateway模式）
4. **状态转移确定性** - 每一步都在可验证的状态空间内

**与Claude Code的关键区别：**
- 状态空间架构不是"让LLM遵守规则"，而是"让LLM物理上无法违反规则"
- 这是架构级别的解决方案，不是Prompt工程的改进

#### 核心问题的新洞察：Claude Code的根本缺陷

**缺陷5：Agent架构的复杂性失控**

Claude Code的Agent架构包含多个组件：
- Lead Agent: 主控Agent
- Sub-agents: 子任务Agent
- MCP Servers: 外部服务集成
- Custom Commands: 自定义命令
- Hooks: 前后置钩子
- Agent SDK: 完全自定义的Agent构建

**问题分析：**
```
多层嵌套的Agent架构 → 状态空间指数级膨胀
     ↓
无法追踪完整状态转移
     ↓
调试困难，行为不可预测
```

**状态空间架构的应对：**
- 显式状态空间定义，状态数量有限
- 状态转移规则明确、可追踪
- 组合式安全：小型、可验证组件

#### 学术前沿：状态空间架构的理论基础

**从搜索理论角度：**
- LLM作为启发式搜索函数（已在方向8中深入分析）
- 状态空间作为搜索空间的数学抽象
- 类型系统作为搜索空间的约束表达

**从形式化方法角度：**
- Refine4LLM: 程序精化作为状态空间约束
- XGrammar: 词法约束作为状态空间定义
- 状态空间架构整合以上方法

#### 实践对比：典型场景分析

| 场景 | Claude Code方式 | 状态空间架构方式 |
|------|-----------------|-----------------|
| **代码生成** | LLM理解Prompt → 生成代码 → 测试验证 | 类型约束 → 状态导航 → 编译检查 |
| **安全审查** | LLM理解安全规则 → 静态分析 | 形式化规约 → 模型检查 |
| **Bug修复** | LLM理解Bug描述 → 尝试修复 → 测试 | 错误状态空间 → 精化步骤 |
| **代码重构** | LLM理解重构目标 → 生成代码 | 重构规则 → 状态转移验证 |

#### 关键洞察总结

**五大根本缺陷：**

1. **Prompt依赖的脆弱性** - "软约束"无法防止LLM的"创造性"
2. **事后验证的低效性** - TDD是补丁，非架构解决方案
3. **状态的黑盒特性** - 决策过程不可追踪
4. **工具能力的失控风险** - 动态权限控制无法穷举危险
5. **Agent架构的复杂性失控** - 多层嵌套导致状态空间膨胀

**状态空间架构的五大优势：**

1. **硬性边界** - 编译期/入口期强制约束
2. **事前验证** - 生成前约束输出空间
3. **状态可追踪** - 显式状态空间，转移可审计
4. **权限类型化** - 权限作为类型，编译期检查
5. **组合式安全** - 小型组件，系统级保证
