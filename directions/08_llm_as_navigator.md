## 研究历程

### 2026-03-10 16:00 LLM导航器深度研究（第二轮）

#### 核心问题
LLM作为启发式函数的理论基础?

#### 研究发现

**1. 搜索算法视角下的LLM**

将LLM视为搜索过程中的启发式函数评估器：
- **状态空间**：代码生成任务中所有有效的代码状态集合
- **动作空间**：可执行的代码编辑操作（类型系统约束下的合法操作）
- **启发式函数**：LLM评估当前状态到目标状态的"距离"
- **搜索策略**：A*、MCTS或其他搜索算法引导LLM探索

**2. Plan Tokens的实践验证（COLM 2024）**

论文 "Guiding Language Model Reasoning with Planning Tokens" (arXiv:2310.05707) 提出了：
- 在每个推理步骤开始时生成"规划token"
- 规划token作为高层计划，引导当前推理步骤
- 只增加0.001%的可训练参数
- 在数学问题和多跳QA数据集上显著提升准确率

这验证了：LLM可以超越"生成器"，扮演"规划/导航"角色。

**3. 导航器模式的架构优势**

| 维度 | 生成器模式 | 导航器模式 |
|------|-----------|-----------|
| **输出** | 完整解决方案 | 下一步最优动作 |
| **正确性保证** | 事后验证 | 事前约束（类型系统） |
| **搜索深度** | 单次生成 | 多步搜索 |
| **可回溯性** | 困难 | 易于实现 |
| **状态管理** | 黑盒 | 显式状态空间 |

**4. LLM作为学习型启发式函数**

传统A*搜索依赖手工设计的启发式函数h(n)：
- 需要领域专家设计
- 静态、无法适应新任务

LLM作为启发式函数的潜在优势：
- 通过预训练获得广泛领域知识
- 动态评估不同路径的可行性
- 可根据上下文调整搜索方向

**5. 与状态空间架构的结合**

在状态空间架构中，LLM导航器的工作模式：

```
用户意图 → 类型约束定义的状态空间 → LLM作为启发式函数评估动作序列 → 执行最优动作
                    ↓                                    ↓
            硬性边界保证有效状态                  LLM只选择"合法"动作
```

关键机制：
1. **状态表示**：类型安全的代码状态
2. **动作空间**：类型系统允许的编辑操作
3. **启发式评估**：LLM评估动作序列的优劣
4. **搜索执行**：确定性系统执行选定的动作
5. **状态验证**：每步执行后验证状态有效性

### 2026-03-10 16:00 LLM导航器深度研究（第三轮 - 搜索算法框架）

#### 核心问题
LLM作为启发式函数的理论基础?

#### 研究发现

**1. Tree of Thoughts (ToT) - 搜索框架的里程碑**

NeurIPS 2023 论文 "Tree of Thoughts: Deliberate Problem Solving with Large Language Models" (arXiv:2305.10601) 提出了：
- **核心思想**: 将CoT扩展为树结构搜索
- **关键机制**:
  - Thought (思维单元): 离散的文本片段，作为推理中间步骤
  - 探索多个推理路径，而非单一链
  - 自我评估选择，支持前瞻和回溯
- **实验结果**:
  - Game of 24: GPT-4 + CoT 仅4% → ToT达到74%
  - Creative Writing, Mini Crosswords 均有显著提升

**2. Plan Tokens - 规划token的层次化推理**

论文 "Guiding Language Model Reasoning with Planning Tokens" (COLM 2024) 提出了：
- 在每个推理步骤开始时生成"规划token"
- 规划token作为高层计划，引导当前推理步骤
- 只增加0.001%的可训练参数
- 在数学问题和多跳QA数据集上显著提升准确率

这验证了：LLM可以超越"生成器"，扮演"规划/导航"角色。

**3. 搜索算法与LLM的结合模式**

| 搜索算法 | LLM角色 | 状态空间架构价值 |
|----------|---------|-----------------|
| **A*** | 启发式函数h(n) | 评估代码状态到目标的"距离" |
| **MCTS** | Rollout策略 | 选择最优编辑动作序列 |
| **Beam Search** | Scorer | 排序Top-K候选代码 |
| **ToT** | 探索+评估 | 多路径推理选择 |

**4. LLM导航器与状态空间架构的数学映射**

```
状态空间定义:
  S = { valid_code_states }     // 类型系统约束的有效代码状态
  A = { edit_operations }       // 合法的代码编辑操作
  T: S × A → S                  // 状态转移函数（确定性）

搜索问题:
  找到路径 p = (s0, a1, s1, a2, ..., sk) 使得:
    - s0 ∈ S (初始状态)
    - sk ∈ S (目标状态，满足类型约束)
    - ∀i: ai ∈ A(si) (动作在当前状态下合法)
    - minimize: cost(p) = Σ g(si, ai) + h(si, target)

LLM作为启发式函数:
  h(s, target) = LLM.score(s, target)  // 评估当前状态到目标的"接近度"
```

**5. 与状态空间架构的深度整合**

在状态空间架构中，LLM导航器的完整工作流：

```
用户意图 → 类型约束定义状态空间S → LLM作为启发式函数
                    ↓                              ↓
            硬性边界保证s∈S              LLM评估h(s, target)
                    ↓                              ↓
            确定性系统执行                搜索算法选择最优动作
                    ↓                              ↓
            每步验证状态有效性            状态转移 T(s, a) → s'
```

关键机制：
1. **状态表示**: 类型安全的代码状态（Rust类型系统）
2. **动作空间**: 类型系统允许的编辑操作
3. **启发式评估**: LLM评估动作序列的优劣
4. **搜索执行**: 确定性系统执行选定的动作
5. **状态验证**: 每步执行后验证状态有效性

#### 架构洞察补充

**LLM导航器的数学基础:**

1. **可容许性 (Admissibility)**
   - 传统A*: h(n) ≤ h*(n) 保证找到最优解
   - LLM作为启发式: 需要约束LLM的"乐观程度"
   - 状态空间架构的优势: 硬性边界防止"过度乐观"

2. **一致性 (Consistency)**
   - 启发式函数的一致性: h(s) ≤ cost(s→s') + h(s')
   - LLM的挑战: 预训练知识可能导致不一致评估
   - 解决方案: 限制LLM在"选择题"模式，减小搜索空间

3. **计算复杂度**
   - 状态空间大小: |S| 可能巨大
   - 动作空间: |A(s)| 取决于类型约束粒度
   - 导航器优势: 类型系统大幅减少无效状态

3. **组合式安全性**
   - 类型系统保证状态有效性
   - API边界约束动作空间
   - LLM只做"选择题"，不做"填空题"

## 架构洞察

### LLM 作为导航器的范式转换
**传统模式（生成器）:**
- LLM直接生成完整解决方案
- 通过Prompt工程约束输出
- 难以保证正确性

**新范式（导航器）:**
- LLM在预定义的状态空间中搜索路径
- 每一步选择都受限于硬性边界
- 搜索过程可验证、可回溯

### 理论基础
1. **启发式函数** —— LLM评估状态空间的搜索方向
2. **A*搜索** —— 结合代价函数和启发式函数的最优路径搜索
3. **多步规划** —— LLM作为高层规划器，执行由确定性系统完成

### 导航器模式的核心价值

1. **分离决策与执行**

   - 避免LLM直接操作产生的错误

2. **可验证的搜索过程**
   - 每步选择都有明确的状态转移
   - 搜索路径可回溯、可审计
   - 失败快速：无效动作入口拒绝

3. **组合式安全性**
