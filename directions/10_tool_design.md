### 2026-03-10 20:00 深度研究（与Claude Code对比）
**核心问题**: 如何设计"无法产生错误"的工具? 与现有AI工具的本质区别是什么?

**研究发现**:
1. **Claude Code权限系统的本质分析**:
   - 观察其源码: 基于React的权限请求组件架构
   - `BasePermissionRequest`: 统一的基础组件处理通用UI
   - `FallbackPermissionRequest`: 兜底权限请求
   - 工具级别: 细粒度到单个工具的权限控制 (`allowedTools` CLI flag)
   - **核心洞察**: 这是典型的"软约束"实现——权限系统在运行时阻止操作，但LLM仍然可以尝试任何操作

2. **硬边界 vs 软约束的工程实现对比**:
   | 特性 | Claude Code (软约束) | 状态空间 (硬边界) |
   |------|---------------------|------------------|
   | 错误处理 | 运行时权限请求 | 编译期类型约束 |
   | LLM行为 | "不应该"做某事 | 物理上"不可能"做某事 |
   | 边界突破 | prompt注入可绕过 | API物理隔离 |
   | 验证时机 | 执行时 | 编译时 |
   | 错误恢复 | 需要用户审批 | 不可能发生 |

3. **Kani Model Checker的验证价值**:
   - bit-precise: 精确到bit级别的验证
   - 自动检查undefined behavior、panics、溢出
   - Function contracts: `#[requires(...)]`, `#[ensures(...)]`
   - 使用 `kani::any()` 创建非确定性输入，自动遍历所有可能值
   - **关键价值**: 验证状态空间工具的**所有可能输入**是否满足规范

4. **无缺陷工具的核心设计模式**:
   ```rust
   // 1. 状态编码到类型 - 无效状态编译期不可能
   struct Tool<S: ToolState> { _state: PhantomData<S> }
   
   // 2. 线性类型 - 状态转换不可逆
   fn execute(self) -> TransitionResult<Output> { ... }
   
   // 3. 纯函数 - 无副作用 = 零状态依赖
   trait PureTool { fn execute(&self, input: In) -> Out; }
   
   // 4. API白名单 - 物理隔离
   struct APIBoundary<T> { whitelist: &'static [&'static str] }
   ```

5. **与现有工具的根本区别**:
   - GitHub Copilot: 概率生成 → 可能出错
   - Claude Code: 权限控制 → 运行时阻止
   - 状态空间架构: **编译期约束 → 物理上不可能出错**

**架构洞察**:
- **零错误的本质**: 不是"减少错误"，而是"错误在工程上不可能发生"
- **工具设计的范式转移**: 从"信任LLM"到"信任系统"
- **渐进式披露的实际价值**: 简单任务暴露少且安全的工具，复杂任务需显式提升权限
- **运行时权限 vs 编译期约束**: 前者是"门"，后者是"墙"

## 架构洞察

### 无缺陷工具设计原则
1. **纯函数** —— 相同的输入总是产生相同的输出，无副作用
2. **副作用隔离** —— IO操作与业务逻辑分离
3. **不变量** —— 系统设计时就不允许非法状态存在
4. **失败快速** —— 错误在最早可能点被捕获

### 工具链设计
- CLI设计：命令行接口作为状态空间的入口
- 配置验证：配置加载时即验证合法性
- 渐进式披露：简单任务简单，复杂任务可行

### 硬边界 vs 软约束（核心区分）
- ❌ 软约束："请你不要修改这个文件"（AI可能不听）
- ✅ 硬边界：API不提供修改该文件的能力（AI物理上做不到）

### 现有AI编码工具的缺陷分析
1. **GitHub Copilot**：基于LLM的概率生成，可能产生错误代码
2. **Claude Code**：基于权限系统的信任模式，依赖"理解"而非强制约束
3. **Amazon Q Developer**：agentic能力，但同样基于LLM概率生成

### 状态空间架构的工具设计思路
- **不信任LLM的理解能力**，通过类型系统和API边界强制约束
- **工具只能是纯函数**：输入→状态转换→输出，无副作用
- **边界即法律**：LLM只能操作受限API，物理上接触不到危险操作

### 2026-03-10 20:24 深度研究（权限层次与最小权限原则）

**核心问题**: 如何将工具权限约束从"软约束"升级为"硬边界"?

**研究发现**:

1. **MiniScope (arXiv:2512.11147, cs.CR, Dec 2025)**:
   - 提出最小权限框架用于工具调用Agent
   - 核心方法：重构工具调用之间的权限层次关系
   - 结合"移动端权限模型"（一次性授权、用时请求）
   - 性能开销：仅1-6%（相比vanilla工具调用Agent）
   - **关键限制**：运行时权限检查，依赖策略文件——属于软约束
   - **与状态空间的区别**：MiniScope基于运行时策略，无法防止类型级别的滥用

2. **DRIFT (arXiv:2506.12104, NeurIPS 2025)**:
   - 动态规则 + 注入内容隔离机制
   - 防御间接prompt注入：外部数据中包含的恶意指令
   - 方法：在工具输出与LLM指令之间建立隔离层
   - **关键限制**：仍然是"检测+防御"模式，不是物理隔离

3. **VIGIL（验证前提交）**:
   - Verify-before-commit：工具执行前验证
   - 检测工具输出流中的注入内容
   - **关键限制**：依赖LLM理解工具输出，可被混淆攻击绕过

4. **ATLASS (arXiv:2503.10071)**:
   - 闭环框架：动态按需生成工具
   - 关键风险：动态生成的工具可能绕过权限系统
   - 人工反馈作为安全保障（软约束！）

5. **状态空间架构的核心突破**:
   
   上述工作都在"软约束"范畴内。状态空间架构的关键创新：
   
   ```
   软约束链:                         硬边界:
   MiniScope策略文件                  Rust类型系统
      ↓ 可被绕过                      ↓ 物理不可能
   运行时权限检查                    编译期权限约束
      ↓ 1-6%开销                      ↓ 零成本抽象
   工具执行前检查                    工具调用时类型验证
   ```
   
   核心实现：`ToolToken<P>` 将权限级别 `P` 编码进类型，错误权限的令牌在编译时就报错。

6. **渐进式权限提升的正确实现**:
   - 默认最小权限（ReadPermission）
   - 提升权限需要显式 `ApprovalToken`（用户审批）
   - 类似OAuth的作用域（`TaskScope<P>`），但由类型系统而非服务器强制执行
   - 关键：`ApprovalToken` 无法通过LLM指令伪造，只能由用户操作生成

7. **LLM集成的关键挑战**:
   - LLM输出JSON字符串，JSON不携带Rust类型信息
   - 解决思路：`LLMDispatcher<P>` 持有令牌，LLM只能请求工具ID字符串
   - 分发层负责将JSON请求映射到已授权的工具令牌
   - 物理隔离：LLM无法直接访问令牌，只能通过受控分发层请求工具

**架构洞察**:
- **"权限层次"的两种实现**: MiniScope（运行时策略文件）vs 状态空间（编译期类型层次）
- **零信任架构的正确实现**: 不信任LLM请求，只信任类型系统的静态证明
- **成本比较**: MiniScope 1-6%运行时开销 vs 状态空间 0%运行时开销（编译期验证）
- **安全性比较**: MiniScope可被策略绕过 vs 状态空间无法绕过（物理约束）

**新增核心资源**:
- MiniScope (arXiv:2512.11147) - 最小权限框架，对比参照
- DRIFT (arXiv:2506.12104, NeurIPS 2025) - 动态规则隔离
- VIGIL - 验证前提交防御
- ATLASS (arXiv:2503.10071) - 动态工具生成

**代码草稿**: `drafts/20260310_2024_tool_permission_hierarchy.rs`
- 权限层次类型系统（PermissionNode trait层次）
- ToolToken<P> 编译期权限绑定
- LLMDispatcher<P> 权限级别的分发器
- TaskScope<P> 任务作用域
- ApprovalToken 用户审批机制

- [ ] Rust类型系统能否完整表达现实世界的权限需求？（尤其是动态权限组合）
- [ ] LLMDispatcher的分发层开销是否可接受？（理论上接近0，需实测）
- [ ] 如何处理跨工具的权限依赖？（工具A需要工具B的输出作为前提）

### 2026-03-10 21:50 深度研究（最新学术进展补充）

**核心问题**: 最新学术研究如何解决Agent工具安全问题？状态空间架构相比有何优势？

**研究发现**:

1. **Traversal-as-Policy (arXiv, March 2026)**:
   - 使用日志蒸馏的门控行为树作为外部化、可验证的策略
   - 核心思想：将LLM的"推理"外部化为可验证的行为树
   - 优点：策略可审计、可验证、可控制
   - **与状态空间的区别**：仍然是"软约束"，依赖行为树的节点执行顺序，而非编译期类型约束

2. **MCP Caller Identity Confusion (arXiv, March 2026)**:
   - 首次系统性研究MCP协议中的调用者身份混淆问题
   - 核心漏洞：MCP客户端/服务器之间的身份验证机制缺失
   - 攻击向量：恶意MCP服务器可伪装成可信工具
   - **与状态空间的区别**：状态空间通过类型系统隔离，物理上不可能伪造身份

3. **Governance Architecture for Autonomous Agent Systems (arXiv, March 2027)**:
   - Agent系统治理框架：威胁、框架、工程实践
   - 核心威胁：prompt注入、检索投毒、不受控制的工具调用
   - **与状态空间的区别**：状态空间将威胁建模为类型系统的非法状态，在编译期排除

4. **状态空间架构 vs 学术前沿对比**:

   | 方案 | 约束机制 | 验证时机 | 可绕过性 |
   |------|---------|---------|---------|
   | Traversal-as-Policy | 行为树执行顺序 | 运行时 | 可能（LLM可跳过节点）|
   | MCP安全方案 | 身份验证协议 | 运行时 | 可能（协议漏洞）|
   | 治理框架 | 规则引擎 | 运行时 | 可能（规则覆盖不全）|
   | **状态空间** | **Rust类型系统** | **编译期** | **物理不可能** |

5. **AgentSentry (arXiv, Feb 2026)**:
   - 通过时序因果诊断和上下文净化减轻间接Prompt注入
   - 方法：检测工具输出中的注入内容，在传递给LLM前净化
   - **关键限制**：仍然是"检测+防御"模式，依赖模式匹配，可被绕过

6. **Silent Egress (arXiv, Feb 2026)**:
   - 研究隐式Prompt注入如何让LLM Agent无声泄露信息
   - 攻击方式：通过工具输出嵌入隐蔽指令，Agent在不知情情况下执行
   - **关键洞察**：工具输出可能是"特洛伊木马"，传统权限系统无法检测
   - **状态空间防御**：纯函数工具的输出是确定性类型，注入内容无法隐藏

7. **SoK: Agentic Skills (arXiv, Feb 2026)**:
   - 系统化研究Agent技能——超越工具使用
   - 核心发现：现有Agent系统缺乏"技能"的可组合性框架
   - **状态空间贡献**：ToolChain<A,B>类型提供可组合性保证

**架构洞察**:
- **学术界的共同局限**：所有最新研究都在"软约束"范畴内
- **状态空间的独特价值**：编译期类型约束 = 物理上不可能绕过
- **安全范式转移**：从"检测威胁"到"排除非法状态"
- **最新攻击向量的防御**：通过类型系统确保工具输出结构，注入内容无处隐藏

**新增核心资源**:

### 2026-03-10 20:54 深度研究（Claude Code权限系统深入分析）

**核心问题**: Claude Code权限系统的工作原理是什么？与状态空间架构的本质区别？

**研究发现**:

1. **Claude Code权限系统的官方架构**:
   - 三层权限类型：Read-only（无需审批）、Bash（需要审批）、Edit（需要审批）
   - 权限模式：default、acceptEdits、plan、dontAsk、bypassPermissions
   - 规则优先级：deny → ask → allow
   - MCP工具支持：mcp__server_name__tool_name格式
   - **关键限制**：所有权限检查都是运行时行为，依赖AI"理解"规则

2. **Claude Code权限系统的脆弱性分析**:
   ```
   运行时权限检查的漏洞:
   - Prompt注入可绕过权限理解
   - 权限规则模式匹配有绕过风险（如curl URL限制）
   - bypassPermissions可禁用所有检查（危险！）
   - Bash权限规则对命令参数的约束很脆弱
   ```

3. **状态空间架构 vs Claude Code：根本性区别**:
   
   | 特性 | Claude Code | 状态空间架构 |
   |------|------------|-------------|
   | 约束机制 | 运行时权限请求 | 编译期类型约束 |
   | 权限表达 | JSON配置+规则匹配 | Rust类型系统 |
   | 验证时机 | 工具调用时 | 编译时 |
   | 绕过难度 | prompt注入可尝试绕过 | 物理上不可能 |
   | 错误类型 | 运行时PermissionDenied | 编译错误 |
   | 权限粒度 | 命令/文件级别 | 类型级别 |

4. **Claude Code的Hook机制**:
   - PreToolUse hook：工具执行前拦截
   - 可用于自定义权限检查逻辑
   - **本质**：仍然是运行时检查，无法提供硬边界

5. **渐进式披露的正确实现**:
   ```
   Claude Code:                            状态空间:
   ┌─────────────────┐                   ┌─────────────────┐
   │ /permissions    │ ← 用户操作        │ 编译期类型      │ ← 编译器强制
   │ 提升权限        │                   │ 提升权限        │
   └────────┬────────┘                   └────────┬────────┘
            ↓                                    ↓
   ┌─────────────────┐                   ┌─────────────────┐
   │ 运行时权限检查   │ ← 可被绕过        │ 编译期约束      │ ← 物理不可能
   └─────────────────┘                   └─────────────────┘
   ```

6. **关键新洞察：权限系统的范式对比**:
   - **Claude Code范式**：Permission as Configuration（权限即配置）
     - 优点：灵活、用户可控
     - 缺点：依赖理解能力、可被绕过
   
   - **状态空间范式**：Permission as Type（权限即类型）
     - 优点：编译期保证、物理不可能绕过
     - 缺点：灵活性较低、需要类型系统设计

### 2026-03-10 21:00 深度研究（形式化验证与工具安全）

**核心问题**: 如何通过形式化验证确保工具设计的正确性?

**研究发现**:

1. **现有安全框架的共同局限**:
   - MiniScope: 基于运行时策略文件的权限管理，依赖"理解"而非强制
   - DRIFT: 动态规则+注入隔离，但仍属于"检测+防御"模式
   - VIGIL: 验证前提交机制，依赖LLM理解工具输出
   - ATLASS: 动态工具生成，人工反馈作为安全保障
   - **共同缺陷**: 均为"软约束"，可被绕过

2. **形式化验证的独特价值**:
   ```
   验证层次对比:
   ┌─────────────────────────────────────────────────────┐
   │ 运行时检测 (MiniScope/DRIFT)                       │
   │   → 可被绕过 → 1-6%性能开销                        │
   ├─────────────────────────────────────────────────────┤
   │ 静态分析 (TypeScript类型检查)                      │
   │   → 部分绕过 → 0%性能开销                          │
   ├─────────────────────────────────────────────────────┤
   │ 形式化验证 (Kani/Coq/Verus)                       │
   │   → 数学证明 → 0%运行时开销                         │
   └─────────────────────────────────────────────────────┘
   ```

3. **Kani Model Checker的验证能力**:
   - **Function Contracts**: `#[requires(...)]`, `#[ensures(...)]`
   - **非确定性输入**: `kani::any()` 自动遍历所有可能值
   - **覆盖验证**: 检查所有可能的执行路径
   - **关键优势**: 编译期完成验证，运行时零开销

   - **第一层**: 类型系统排除无效状态（日常开发）
   - **第二层**: 属性测试发现边缘情况（CI/CD）
   - **第三层**: 模型检查验证关键属性（关键组件）
   - **第四层**: 定理证明验证完整规格（安全关键）

### 2026-03-10 21:12 深度研究（工具组合与事务性）

**核心问题**: 当多个工具组合使用时，如何保证原子性和一致性?

**研究发现**:

1. **现有AI工具的组合问题**:
   - Claude Code / OpenCode: 工具调用是独立的，依赖LLM理解操作顺序
   - 没有内置的事务机制：如果中间步骤失败，可能留下不一致状态
   - 例如：写入文件A成功，写入文件B失败 → 需要手动回滚
   - **核心缺陷**: 缺乏"全部成功或全部失败"的保证

2. **状态空间架构的工具组合模型**:
   ```
   传统方式:                          状态空间:
   ┌──────┐  ┌──────┐  ┌──────┐    ┌─────────────────────┐
   │Tool A│→ │Tool B│→ │Tool C│    │ ToolChain<A, B, C>  │
   └──────┘  └──────┘  └──────┘    │  - 类型安全组合      │
        ↓       ↓       ↓           │  - 事务性保证        │
   [成功]   [失败]   [未执行]       │  - 回滚自动化        │
   需要手动回滚                  └─────────────────────┘
   ```

3. **Rust类型系统实现工具组合**:
   ```rust
   // 工具链类型 - 编译期保证组合安全性
   pub struct ToolChain<First, Second> { ... }
   
   // 组合操作符
   impl<First: Tool, Second: Tool> Compose for ToolChain<First, Second> 
   where First::Output: InputFor<Second>
   {
       fn execute(&self, input: First::Input) -> TransactionResult<First::Output, Second::Output> {
           // 1. 执行第一个工具
           let first_result = self.first.execute(input);
           
           // 2. 检查结果，决定是否继续
           match first_result {
               Ok(output) => {
                   // 3. 转换为第二个工具的输入
                   let second_input = transform(output);
                   // 4. 执行第二个工具
                   self.second.execute(second_input)
               }
               Err(e) => TransactionResult::Rollback(e)
           }
       }
   }
   
   // 事务结果类型
   pub enum TransactionResult<A, B> {
       Complete(Ok(B)),
       Partial(Ok<A>, Err),    // 第一个成功，第二个失败
       RolledBack(Err)         // 第一个失败，已回滚
   }
   ```

4. **工具组合的类型约束**:
   ```rust
   // 输出类型必须匹配输入类型
   pub trait InputFor<Next: Tool> {
       type Output;
   }
   
   // 文件读取 → 文件写入 (类型匹配)
   impl InputFor<WriteFileTool> for ReadFileOutput {
       type Output = WriteFileInput;
   }
   
   // 编译 → 测试 (类型不匹配，编译期错误)
   // impl InputFor<TestTool> for ReadFileOutput { ... } // 编译失败！
   ```

5. **回滚机制的类型安全实现**:
   ```rust
   // 可回滚操作 trait
   trait Rollbackable {
       type RollbackAction;
       fn rollback(self) -> Self::RollbackAction;
   }
   
   // 文件写入的回滚：删除文件
   impl Rollbackable for FileWriteAction {
       fn rollback(self) -> FileDeleteAction {
           FileDeleteAction { path: self.path }
       }
   }
   
   // 事务执行器保证回滚
   fn execute_with_rollback<T: Rollbackable>(tool: T) -> Result<T::Output, T::RollbackAction> {
       let result = tool.execute();
       if result.is_err() {
           // 自动回滚
           tool.rollback();
       }
       result
   }
   ```

6. **与现有方案的根本区别**:
   | 特性 | Claude Code | 状态空间架构 |
   |------|------------|-------------|
   | 组合方式 | LLM决定调用顺序 | 类型系统约束组合 |
   | 事务性 | 无 | 编译期保证 |
   | 回滚 | 手动/无 | 自动类型安全 |
   | 错误传播 | 依赖LLM理解 | 类型系统强制 |
   | 可组合性验证 | 运行时发现 | 编译期报错 |

7. **工具组合的实际价值**:
   - **代码重构场景**: 读取文件 → 分析 → 写入新文件 → 删除旧文件
   - **批量操作场景**: 批量读取 → 批量处理 → 批量写入
   - **测试场景**: 编译 → 运行测试 → 收集结果 → 报告
   - 所有这些场景都需要原子性保证

**架构洞察**:
- **组合安全 = 类型安全**: 工具组合的可组合性在编译期验证
- **事务性是默认值**: 不是"可选特性"，而是"设计原则"
- **回滚即类型**: 回滚操作也是类型系统的一部分
- **渐进式复杂性**: 简单任务用简单工具链，复杂任务用事务性工具链

### 2026-03-11 09:55 深度研究（最新Agent安全攻击向量分析）

**核心问题**: 最新的Agent安全攻击如何突破现有防御？状态空间架构如何应对？

**研究发现**:

1. **Silent Egress (arXiv, Feb 2026)** - 隐式Prompt注入导致信息泄露:
   - 攻击方式：通过工具输出嵌入隐蔽指令，Agent在不知情情况下执行
   - 关键发现：工具输出可能是"特洛伊木马"
   - 状态空间防御：纯函数工具的输出是确定性类型，注入内容无法隐藏

2. **MCP Security Bench (MSB) (arXiv, Oct 2025)** - MCP协议攻击基准:
   - 系统性研究MCP协议的安全漏洞
   - 攻击向量：工具投毒、对抗性攻击
   - 状态空间防御：类型系统隔离，物理上不可能伪造工具身份

3. **Zombie Agents (arXiv, Mar 2026)** - 持久化控制攻击:
   - 攻击方式：通过自我强化注入实现对Agent的持久控制
   - 关键洞察：现有防御无法防止Agent自我修改
   - 状态空间防御：线性类型防止状态被恶意修改

4. **AgenTRIM (arXiv, Jan 2026)** - 工具风险缓解:
   - 系统性分析Agent工具调用风险
   - 防御方法：风险评估 + 缓解策略
   - 状态空间防御对比：状态空间在编译期排除所有风险操作

5. **AgentSys (arXiv, Feb 2026)** - 显式层次记忆管理:
   - 防御间接prompt注入
   - 方法：通过层次记忆管理隔离外部输入
   - 状态空间防御对比：类型系统提供更强的隔离保证

6. **STELP (arXiv, Jan 2026)** - LLM生成程序的安全执行:
   - 安全转译和执行LLM生成的代码
   - 方法：沙箱 + 验证 + 执行分离
   - 状态空间防御对比：编译期类型约束比运行时沙箱更安全

7. **最新攻击向量的类型系统防御**:

   | 攻击类型 | 现有防御 | 状态空间防御 |
   |---------|---------|-------------|
   | 隐式注入 | 输出净化 | 确定性类型输出 |
   | 工具投毒 | MCP验证 | 编译期工具验证 |
   | 持久化控制 | 运行时监控 | 线性类型防修改 |
   | 资源耗尽 | 限流 | 编译期资源约束 |
   | Agent间协作攻击 | 隔离策略 | 类型级别隔离 |

8. **Agent工具安全的未来趋势**:
   - 从"检测+防御"到"排除非法状态"
   - 从"运行时验证"到"编译期保证"
   - 从"单一工具"到"工具链组合安全"
