# 07_layered_design

## 方向名称
分层设计：四层确定性三明治架构

## 核心问题
Syntax→Semantic→Pattern→Domain 如何转换?

## 研究历程

### 2026-03-10 14:49 深度研究：形式语言理论与状态机视角

**理论基础 - 正则语言与状态机类比**:

四层架构与形式语言理论中的**Chomsky层级**存在有趣的对应关系：

| Chomsky层级 | 状态空间架构层级 | 自动机模型 | 表达能力 |
|-------------|-----------------|-----------|---------|
| 正则语言 (L3) | Domain (业务逻辑) | 有限状态机 | 有限状态 |
| 上下文无关语言 (L2) | Pattern (设计模式) | 下推自动机 | 栈内存 |
| L1 Semantic | 类型系统 | 带类型的状态机 | 有限状态+类型 |
| L0 Syntax | Token序列 | 有穷自动机 | 线性 |

**关键洞察 - 状态转移的确定性保证**:

1. **正则语言的正则性** → 业务逻辑必须是确定性的
   - 状态空间中的每个"状态"实际上是一个**正则状态**
   - 状态转移函数必须是**全函数**（ total function）
   - 每个输入必须产生唯一输出

2. **泵引理与状态空间边界**
   - 状态空间是**有界的**（类比正则语言）
   - 无限状态空间 → 需要引入新层（L3→L4?）
   - 这解释了为什么简单prompt难以控制LLM：它工作在**上下文无关语言**层级

3. **编译器前端技术的直接应用**
   - **Lexer**: LLM输出 → TokenStream (已由XGrammar实现)
   - **Parser**: TokenStream → AST (CFG解析)
   - **Type Checker**: AST → TypedAST (Rust类型系统)
   - **Code Generator**: TypedAST → 可执行代码

**状态空间架构的数学形式化**:

```
StateSpace = (S, s₀, Σ, δ, F)
其中:
- S: 状态集合 (有限或可数无限)
- s₀: 初始状态
- Σ: 输入字母表 (LLM可用的动作空间)
- δ: S × Σ → S (确定性转移函数)
- F: 接受状态集合 (有效完成状态)
```

**关键约束**:
- **δ必须是全函数**: 每个状态-动作对必须有明确定义的后续状态
- **不接受状态**: δ(s, a) = ⊥ 表示无效动作
- **不接受状态的处理**: 立即失败，而不是传递到下一层

**与Reed-Solomon码的类比（信息论视角）**:

- Reed-Solomon: 通过冗余检测和纠正错误
- 状态空间: 通过**结构化冗余**（多层转换）确保正确性
- 两者都使用**代数结构**确保确定性

**下一步研究方向**:
1. 设计L1→L2的"类型化模式匹配"转换器
2. 研究L2 Pattern库的**完备性证明**（覆盖80%常见设计模式）
3. 构建"状态转移矩阵"的Rust数据结构
4. 测量实际系统中的"无效状态拒绝率"

### 2026-03-10 14:00 深度研究：层间转换的工程实现

**核心问题**: 如何确保 Syntax→Semantic→Pattern→Domain 的每一步转换都是确定性的,且转换失败能被正确捕获?

**新洞察 - 转换失败的三种模式**:

| 失败模式 | 发生层级 | 捕获机制 | 恢复策略 |
|----------|----------|----------|----------|
| 语法错误 | L0→L1 | CFG解析器 | 拒绝LLM输出,重新生成 |
| 类型错误 | L1→L2 | Rust类型检查器 | 类型推导修正或降级 |
| 模式不匹配 | L2→L3 | DSL解释器 | 回退到默认模式 |

**关键技术突破 - 零拷贝转换路径**:

传统流程: LLM生成Token → L0解析 → L1类型检查 → L2模式匹配 → L3执行
问题: 每层都可能失败,错误信息逐层丢失

新架构流程:
```
LLM输出 → [XGrammar约束解码] → 类型化TokenStream 
       → [Rust类型推断] → TypedAST 
       → [模式匹配宏] → DomainAction 
       → [DSL解释器] → 执行结果
```

关键: **每层转换都是TOTAL函数**,即:
- 输入合法 → 输出唯一确定
- 输入非法 → 立即报错,不产生下游输出

**与现有LLM编码工具的根本差异**:

| 特性 | Claude Code | 状态空间架构 |
|------|-------------|--------------|
| 约束时机 | 生成后验证 | 生成时约束 |
| 错误处理 | 重试机制 | 编译期排除 |
| LLM角色 | 生成器(可能出错) | 导航器(只做选择) |
| 边界类型 | 软约束(prompt) | 硬边界(类型系统) |

**Rust类型系统的关键作用**:

1. **线性类型**: 保证状态不可克隆,只能转移(防止状态逃逸)
2. **过程宏**: L2→L3转换的编译期展开,无运行时开销
3. **TypeState模式**: 状态作为类型,编译期保证状态转换合法

**下一步研究方向**:
1. 设计L0→L1的"类型化TokenStream"数据结构
2. 实现L1→L2的"模式选择器"类型类
3. 构建L2→L3的"领域动作"DSL语法树
4. 测量转换失败率 vs 传统方案的错误重试率

### 2026-03-10 14:18 深度研究：编译器理论视角

**理论基础 - 多层IR转换**:

现代编译器(如LLVM, GCC)的多层IR设计为状态空间架构提供了重要参考:

| 编译器层级 | 状态空间架构对应 | 关键特性 |
|-----------|-----------------|---------|
| Token Stream | L0 Syntax | 线性字符序列 |
| Lexer Output (LLVM Lexer) | L0→L1 桥接 | 词法单元+位置信息 |
| AST | L1 Semantic | 层次结构+类型标注 |
| MIR (Rust) | L2 Pattern | 中间表示+数据流 |
| LIR | L3 Domain | 目标代码生成 |

**关键洞察 - 编译器vs状态空间架构**:

1. **SSA (Static Single Assignment) 形式**
   - 传统编译器用SSA简化数据流分析
   - 状态空间架构可用SSA简化状态转换验证
   - 每个状态变量只能赋值一次，消除隐式依赖

2. **CPS (Continuation Passing Style)**
   - 控制流显式化
   - 状态空间架构可借鉴：将Domain逻辑转为CPS，使得每个"状态"都是 continuation
   - 好处：控制流错误变成类型错误

3. **Def-Use Chain**
   - 编译器追踪每个变量的定义和使用位置
   - 状态空间架构可追踪状态依赖图
   - 好处：状态逃逸检测 = 编译期错误

**与CompCert的对比**:

CompCert是经过形式化验证的C编译器，其多层IR转换路径:
```
C源程序 → Parsed AST → Typed AST → Cminor → CminorSel → RTL → LTL → Linear → Mach
```
每层之间的转换都经过Coq证明保持语义一致性。

**状态空间架构可借鉴的设计**:
- 每层转换都必须是**语义保持的** (semantics-preserving)
- 转换函数本身可以用形式化方法验证
- 错误在层间传播时有清晰的错误类型

**待验证假设**:
- [ ] L0→L1零拷贝路径的可行性（避免生成后解析）
- [ ] L2 Pattern库的完备性（覆盖80%常见设计模式）
- [ ] 逃逸尝试率是否真正为0 
通过分析编译器理论与抽象解释框架，发现现有LLM约束方案（XGrammar、Type-Constrained等）的关键缺陷：**它们仅在单一抽象层施加约束，缺乏层间转换的硬性边界**。

**关键洞察**:
**LLM应被视为"分层状态空间中的导航器"，而非"从噪声到信号的生成器"**。每一层提供闭合的操作集合，层间跃迁由编译器/类型系统强制执行，不存在提示工程导致的逃逸路径。

**四层确定性三明治架构**:
```
L3 Domain (业务逻辑层)
    ↑↓ [确定性DSL解释器]
L2 Pattern (设计模式层) 
    ↑↓ [确定性宏/模板展开]
L1 Semantic (类型/语义层)
    ↑↓ [确定性类型检查]
L0 Syntax (Token序列层)
    ↑↓ [确定性CFG解析]
```

**关键设计**:
- **层内**: LLM作为导航器在受限子空间中选择路径（如选择哪个设计模式）
- **层间**: 100%确定性转换，由编译期检查保证，LLM无法干预
- **失败模式**: 无效层间转换在入口被拒绝（Parse Error/Type Error），绝不传递到下一层

**与现有方案对比**:
相比Claude Code的"生成后验证"，此架构将**验证成本从运行时移至编译期**，且LLM始终操作在"已验证的上下文"中，而非自由生成后修剪。

**下一步研究方向**:
1. 实现L0→L1的确定性桥梁：基于XGrammar扩展，构建从约束解码直接到类型化AST的零拷贝路径
2. L2 Pattern库设计：研究常见算法模式的形式化规约，使LLM选择具备可证明的完备性
3. 分层错误反馈机制：错误通过层级向上传播，在适当的抽象层进行修复尝试
4. 与Praetorian架构集成：将Thin Agent置于L2层，Fat Platform覆盖L0/L1/L3

**关键验证指标**: 测量LLM在分层架构中的"逃逸尝试率"，目标为0（理论上不可能）

## 架构洞察

### 层间转换特质
层间转换必须是**确定性**的，只有层内导航才允许LLM启发式搜索：

| 层级 | LLM角色 | 确定性组件 |
|------|---------|-----------|
| L0 Syntax | 无（纯约束解码） | XGrammar CFG解析 |
| L1 Semantic | 无（纯类型检查） | Rust类型系统 |
| L2 Pattern | **导航器**（选择设计模式） | 模式匹配引擎 |
| L3 Domain | 无（纯业务逻辑执行） | DSL解释器 |

### 与数据仓库架构类比
数据仓库的ODS→DWD→DWS→ADS分层与状态空间架构高度同构：
- 都是**渐进式精化**（原始→清洗→聚合→应用）
- 都有**硬性边界**（Schema/类型约束）
- 都支持**血缘追踪**（字段级依赖 vs 状态转换边）
- 都具备**错误隔离**（分层故障不级联）
