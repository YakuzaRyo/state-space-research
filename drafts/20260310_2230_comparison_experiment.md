# 对比实验设计：软约束 vs 硬边界

## 实验名称
**AI编码架构对比实验：Claude Code风格（软约束） vs 状态空间架构（硬边界）**

## 实验目标
量化比较两种架构范式在标准代码生成任务集上的表现，验证状态空间架构的核心假设。

## 核心假设

1. **H1**: 硬性边界在复杂任务上的成功率显著高于软约束
2. **H2**: 事前验证的总成本（生成+验证）低于事后验证
3. **H3**: 状态空间架构可显著降低安全漏洞率
4. **H4**: 开发者对状态空间架构的接受度可能较低（学习曲线）
5. **H5**: 混合架构（软约束+硬边界）可能是最优解

## 任务集选择

### 基准数据集
- **HumanEval**: 164个Python函数级编程任务
- **MBPP (Mostly Basic Python Problems)**: 974个Python基础问题
- **SWE-Bench Lite**: 300个真实GitHub Issue修复任务

### 任务分类

| 类别 | 难度 | 数量 | 示例 |
|------|------|------|------|
| 简单函数 | ⭐ | 30 | 字符串处理、数学计算 |
| 中等算法 | ⭐⭐ | 30 | 数据结构操作、排序算法 |
| 复杂类设计 | ⭐⭐⭐ | 20 | 状态机实现、API设计 |
| 安全敏感 | ⭐⭐⭐ | 20 | 用户输入处理、数据库操作 |

## 对照组设计

### 组A: Claude Code风格（软约束）

**配置**:
- 使用自然语言Prompt描述任务
- 依赖LLM自主决策架构和实现
- 使用标准API（无类型约束）
- 事后运行测试验证正确性
- 人工审查安全漏洞

**Prompt示例**:
```
请编写一个Python函数，接收用户输入并查询数据库。
要求：
- 防止SQL注入攻击
- 处理异常情况
- 返回查询结果列表
```

### 组B: 状态空间架构（硬边界）

**配置**:
- 使用类型约束+结构化Prompt
- LLM在受限类型空间中选择
- 强制使用类型安全的API（如参数化查询）
- 编译期/生成期验证
- 类型系统保证基本安全

**Prompt示例**:
```
类型签名: query_db(conn: DatabaseConnection, user_input: SanitizedString) -> Result[List[Row], QueryError]
约束：
- user_input必须经过SanitizedString验证器
- 必须使用参数化查询API
- 错误必须使用Result类型处理
```

### 组C: 混合架构（对照）

**配置**:
- 软约束指导高层架构
- 硬边界保证关键安全属性
- 允许在约束不足时回退到软约束

## 度量指标

### 1. 成功率 (Success Rate)
- **定义**: 通过所有测试用例的任务比例
- **测量**: 自动测试框架运行
- **预期**: B组 > A组（复杂任务）

### 2. 时间效率
- **生成时间 (T_gen)**: 从Prompt到可运行代码的时间
- **验证时间 (T_verify)**: 检查代码正确性所需时间
- **总时间 (T_total)**: T_gen + T_verify
- **预期**: B组的T_verify显著低于A组

### 3. 代码质量
- **静态分析得分**: Pylint/Ruff得分
- **圈复杂度**: Cyclomatic Complexity
- **代码重复率**: DRY原则遵守度
- **预期**: B组在结构质量上更优

### 4. 安全性
- **漏洞扫描**: Bandit安全扫描结果
- **注入攻击测试**: 模拟SQL注入、命令注入
- **权限问题**: 文件系统权限、网络访问
- **预期**: B组漏洞率显著低于A组（目标：<5% vs 40%）

### 5. 开发者理解度
- **代码解释测试**: 开发者解释生成代码的能力
- **修改任务完成时间**: 对生成代码进行小修改的时间
- **调试能力**: 定位并修复bug的能力
- **预期**: B组理解度更高（显式约束）

### 6. 资源消耗
- **LLM API调用次数**: 生成+修正的调用次数
- **Token消耗**: 输入+输出token总数
- **计算成本**: API费用估算
- **预期**: B组调用次数更少（约束减少搜索空间）

## 统计分析方法

### 假设检验
- **显著性水平**: α = 0.05
- **检验方法**:
  - 成功率: 卡方检验
  - 连续变量: t检验或Mann-Whitney U检验
  - 多组比较: ANOVA

### 效应量计算
- **Cohen's d**: 连续变量的效应大小
- **Odds Ratio**: 成功率的效应大小
- **置信区间**: 95% CI

### 样本量计算
- **功效**: 1-β = 0.80
- **效应大小**: 中等效应 (d=0.5)
- **所需样本**: 每组最少64个任务/参与者

## 风险评估

### 技术风险
- **状态空间约束实现复杂度**: 需要定义类型系统和API边界
- **LLM兼容性**: 某些LLM可能不擅长遵循严格类型约束
- **测量偏差**: 验证时间的测量可能受主观因素影响

### 缓解措施
- 使用成熟的类型系统（如Rust或Python的typing）
- 提供清晰的约束文档和示例
- 使用自动化工具测量时间（IDE插件）
