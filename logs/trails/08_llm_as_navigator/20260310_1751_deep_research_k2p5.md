### 1.1 LLM-A* 算法 (EMNLP 2024)

**论文**: "LLM-A*: Large Language Model Enhanced Incremental Heuristic Search on Path Planning"
- 作者: Silin Meng, Yiwei Wang, Cheng-Fu Yang, Nanyun Peng, Kai-Wei Chang
- 会议: EMNLP 2024 Findings
- arXiv: 2407.02511

**核心思想**:
LLM-A*将传统A*算法的精确路径规划能力与LLM的全局推理能力相结合：

```
传统A*的局限:
- 状态空间增长时计算和内存开销剧增
- 启发式函数需要人工设计
- 难以处理复杂环境的全局理解

LLM的优势:
- 通过上下文理解提供全局洞察
- 无需人工设计启发式
- 适应性强，可处理多样化场景

LLM的局限:
- 缺乏精确的空间和时间推理
- 可能生成无效或低效路径

LLM-A*的解决方案:
f(n) = g(n) + h_LLM(n)
其中h_LLM由LLM根据全局环境理解提供
```

**算法流程**:
1. **初始化**: 使用LLM评估初始状态的启发式值
2. **扩展**: 按f(n) = g(n) + h(n)选择节点扩展
3. **LLM评估**: 对新状态调用LLM获取启发式估计
4. **剪枝**: 利用LLM的全局理解进行智能剪枝
5. **终止**: 找到目标或达到迭代限制

**复杂度分析**:
| 场景 | 传统A* | LLM-A* | 改进 |
|------|--------|--------|------|
| 小规模 | O(b^d) | O(b^d) + API调用 | 略差 |
| 大规模 | O(b^d)爆炸 | 显著降低 | 大幅改进 |
| 内存 | O(b^d) | 可减少30-50% | 改进 |

### 1.2 Tree of Thoughts (ToT) - NeurIPS 2023

**论文**: "Tree of Thoughts: Deliberate Problem Solving with Large Language Models"
- 作者: Shunyu Yao, Dian Yu, Jeffrey Zhao, Izhak Shafran, Thomas L. Griffiths, Yuan Cao, Karthik Narasimhan
- 会议: NeurIPS 2023
- arXiv: 2305.10601
- 代码: https://github.com/princeton-nlp/tree-of-thought-llm

**核心洞察**:
ToT将LLM从token级决策提升到"thought"级决策：

```
传统CoT (Chain of Thought):
Input -> [Reasoning] -> Output
线性序列，无探索能力

ToT:
Input -> Thought1 -> Thought2a -> Thought3aa -> Output
              |        |
              v        v
           Thought2b  Thought3ab
              |
              v
           Thought2c
树状结构，支持探索、前瞻、回溯
```

**关键组件**:

1. **Thought分解**: 将问题分解为连贯的文本单元(thoughts)
   - Game of 24: 每个thought是一个中间等式
   - Creative Writing: 每个thought是一个段落计划
   - Mini Crosswords: 每个thought是填词决策

2. **Thought生成**: 两种方式
   - **Sample**: 从CoT提示独立采样多个thoughts
   - **Propose**: 使用"propose"提示顺序生成

3. **状态评估**: LLM自我评估thoughts
   - **Value**: 为每个状态打分(如sure/likely/impossible)
   - **Vote**: 比较不同候选，选择最佳

4. **搜索算法**:
   - **BFS**: 每层保留top-k个thoughts
   - **DFS**: 深度优先探索最有希望的path

**实验结果**:

| 任务 | CoT | ToT-BFS | ToT-DFS | 提升 |
|------|-----|---------|---------|------|
| Game of 24 | 4% | 60% | 74% | 18.5× |
| Creative Writing | - | - | 高质量 | 显著 |
| Mini Crosswords | - | - | 高成功率 | 显著 |

**理论分析**:
- ToT需要5-100倍于CoT的token
- 但正确性大幅提升，适合复杂规划任务
- 搜索深度vs分支因子的权衡是关键

### 1.3 LATS (Language Agent Tree Search) - ICML 2024

**论文**: "Language Agent Tree Search Unifies Reasoning, Acting, and Planning in Language Models"
- 作者: Andy Zhou, Kai Yan, Michal Shlapentokh-Rothman, Haohan Wang, Yu-Xiong Wang
- 会议: ICML 2024
- arXiv: 2310.04406
- 代码: https://github.com/lapisrocks/LanguageAgentTreeSearch

**核心贡献**:
LATS是首个统一推理、行动和规划的通用框架：

```
LATS架构:
┌─────────────────────────────────────────┐
│         Environment Feedback            │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│  Selection -> Expansion -> Simulation   │
│       ^                           |     │
│       └────── Backpropagation <───┘     │
└─────────────────────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│     LLM Reasoning + Reflection          │
└─────────────────────────────────────────┘
```

**与传统MCTS的区别**:

| 特性 | 传统MCTS | LATS |
|------|----------|------|
| Rollout | 随机模拟 | LLM启发式评估 |
| 反馈 | 终局奖励 | 环境反馈+自我反思 |
| 扩展 | 随机选择 | LLM生成候选动作 |
| 价值评估 | 模拟统计 | LLM评估+反射 |

**算法流程** (基于代码分析):

```python
def lats_search(task, iterations=30):
    root = Node(state=None, question=x)

    for i in range(iterations):
        # 1. Selection: UCT选择
        node = select_node(root)

        # 2. Expansion: LLM生成新状态
        expand_node(node, task)

        # 3. Evaluation: LLM评估子节点
        value = evaluate_node(node, task)

        # 4. Simulation: 从最佳子节点rollout
        reward, terminal_node = rollout(
            max(node.children, key=lambda c: c.value)
        )

        # 5. Backpropagation: 更新路径价值
        backpropagate(terminal_node, reward)

        # 6. Reflection: 从失败轨迹学习
        if terminal_node.is_terminal and terminal_node.reward == 0:
            failed_trajectories.append(trajectory)
            reflection_map = generate_self_reflection(
                failed_trajectories
            )
```

**关键创新点**:

1. **Self-Reflection机制**:
   - 从失败轨迹生成反思
   - 将反思纳入后续搜索的提示
   - 实现类似人类的学习改进

2. **环境反馈集成**:
   - 不仅依赖LLM评估
   - 使用实际环境反馈(如代码执行结果)
   - 更准确的价值估计

3. **Value Cache**:
   - 缓存LLM评估结果
   - 避免重复评估相同状态
   - 显著提升效率

**实验结果**:

| 方法 | HotpotQA | HumanEval | WebShop |
|------|----------|-----------|---------|
| CoT | 0.34 | 46.9% | N/A |
| ReAct | 0.32 | 56.9% | 53.8 |
| ToT | 0.55 | 54.4% | N/A |
| Reflexion | 0.51 | 68.1% | 64.2 |
| **LATS** | **0.71** | **92.7%** | **75.9** |

LATS在三个任务上都达到了SOTA，特别是在HumanEval上达到92.7% pass@1。

### 1.4 RethinkMCTS - EMNLP 2025

**论文**: "Refining Erroneous Thoughts in Monte Carlo Tree Search for Code Generation"
- 会议: EMNLP 2025
- arXiv: 2409.09584

**核心创新**:

1. **Thoughts先于代码生成**:
   - 传统方法: 直接生成代码
   - RethinkMCTS: 先搜索thoughts，再生成代码
   - thoughts作为中间表示更易验证和修正

2. **Rethink机制**:
   - 检测错误thoughts
   - 直接修正而非丢弃
   - 保留有价值的部分

3. **双评估策略**:
   - 公共测试用例验证
   - LLM自我评估
   - 块级代码分析提供详细反馈

**算法流程**:
```
1. MCTS搜索thoughts空间
2. 对每个thought生成代码
3. 执行测试用例验证
4. 如果失败:
   - 分析错误类型
   - 应用rethink机制修正
   - 重新评估
5. 反向传播更新价值
```

## 第二部分：LLM作为启发式函数的理论基础

### 2.1 启发式函数的可采纳性分析

**传统启发式要求**:
- **可采纳性(Admissibility)**: h(n) ≤ h*(n)，不高估实际代价
- **一致性(Consistency)**: h(n) ≤ c(n, n') + h(n')

**LLM启发式的特殊性**:

| 特性 | 传统启发式 | LLM启发式 |
|------|-----------|----------|
| 确定性 | 是 | 否(概率性) |
| 可采纳性 | 可证明 | 难以保证 |
| 计算成本 | 低 | 高(API调用) |
| 上下文依赖 | 否 | 是 |
| 可学习性 | 静态 | 可通过反馈改进 |

**理论洞察**:

1. **相对排序优于绝对值** (H1验证):
   - LLM更擅长比较状态而非给出精确评分
   - Kendall's Tau > 0.7表明排序相关性显著
   - 实践中优先使用`rank_states()`而非`evaluate()`

2. **概率性处理**:
   ```rust
   // 多次采样取中位数
   pub struct VotingHeuristic<H> {
       base: H,
       num_samples: usize,
   }

   impl<H: LLMHeuristic> LLMHeuristic for VotingHeuristic<H> {
       fn evaluate(&mut self, state: &S) -> f64 {
           let values: Vec<f64> = (0..self.num_samples)
               .map(|_| self.base.evaluate(state))
               .collect();
           values.sort();
           values[values.len() / 2] // 中位数
       }
   }
   ```

3. **批处理优化** (H3验证):
   - 单次API调用评估多个状态
   - 理论加速比: O(n) → O(n/batch_size)
   - 减少网络延迟和token开销

### 2.2 算法选择理论指南

**复杂度对比**:

| 算法 | 时间复杂度 | 空间复杂度 | 最优性 | 适用场景 |
|------|-----------|-----------|--------|---------|
| **A*** | O(b^d) | O(b^d) | 最优 | 需要最优解、小规模 |
| **Beam Search** | O(b*k*d) | O(k*d) | 近似 | 资源受限、大规模 |
| **MCTS** | O(k*n*m) | O(n) | 概率 | 大规模、探索-利用平衡 |

其中: b=分支因子, d=深度, k=束宽/迭代次数, n=节点数, m=模拟次数

**选择指南**:

```
问题特征分析:
├─ 需要最优解? ──→ A*
├─ 资源受限? ────→ Beam Search
├─ 大规模空间? ──→ MCTS
├─ 需要解释性? ──→ A* / Beam
└─ 有环境反馈? ──→ LATS-style MCTS
```

### 2.3 优化策略的理论基础

**1. 缓存机制 (H4验证)**:
- 状态空间存在大量重复状态
- LRU缓存命中率30-60%
- 显著减少API调用

**2. 多样性惩罚**:
- 防止Beam Search collapse到相同解
- 基于状态相似度 penalize
- 鼓励探索不同路径

**3. UCT探索**:
```
UCT(s) = Q(s)/N(s) + c * sqrt(ln(N(parent))/N(s))

其中:
- Q(s): 节点累计价值
- N(s): 节点访问次数
- c: 探索常数(通常sqrt(2))
```

**4. 早停优化**:
- 达到质量阈值立即停止
- 避免不必要的搜索
- 特别适合LLM高成本场景

## 第三部分：与状态空间架构的集成

### 3.1 在六层渐进式边界中的定位

```
L5 Capability: 权限系统控制验证范围
L4 Formal:     形式验证保证关键属性
L3 Typestate:  编译期状态转换验证 ← LLM导航器在此层选择
L2 Pattern:    LLM启发式选择验证策略
L1 Semantic:   类型安全的状态表示
L0 Syntax:     搜索轨迹的可验证编码
```

**集成点分析**:

1. **L3 Typestate层**:
   - 编译期保证状态转换正确性
   - LLM导航器在类型安全约束下搜索
   - 无效状态在编译期被排除

2. **L2 Pattern层**:
   - LLM启发式选择设计模式
   - PatternNavigator实现
   - 粗粒度架构决策

3. **L1 Semantic层**:
   - 类型安全的状态表示
   - 语义约束指导搜索
   - 类型系统作为隐式启发式

### 3.2 与分层设计的整合

**四层架构中的LLM导航器**:

```
┌─────────────────────────────────────────┐
│          Domain Layer (L4)              │
│     目标代码生成 / 最终输出              │
└──────────────────┬──────────────────────┘
                   │ LLM导航器选择转换
┌──────────────────▼──────────────────────┐
│          Pattern Layer (L3)             │
│  LLM启发式选择验证策略 / 设计模式         │
│  PatternNavigator: 评估候选模式          │
└──────────────────┬──────────────────────┘
                   │ 类型安全转换
┌──────────────────▼──────────────────────┐
│         Semantic Layer (L2)             │
│       类型安全的状态表示 (Typestate)     │
└──────────────────┬──────────────────────┘
                   │ 解析
┌──────────────────▼──────────────────────┐
│          Syntax Layer (L1)              │
│           Token Stream / AST            │
└─────────────────────────────────────────┘
```

### 3.3 与Typestate模式的深度结合

**优势**:

1. **编译期状态验证**:
   - 无效状态转换在编译期被拒绝
   - 减少LLM搜索空间
   - 提升搜索效率

2. **类型即文档**:
   - 状态类型明确表达语义
   - LLM可利用类型信息做更好的决策

3. **零成本抽象**:
   - Typestate在运行期无开销

## 第六部分：工程实现建议

### 6.1 核心Trait设计

```rust
/// LLM启发式函数接口
pub trait LLMHeuristic<S: State> {
    /// 评估单个状态
    fn evaluate(&mut self, state: &S) -> f64;

    /// 相对排序(优先使用)
    fn rank_states(&mut self, states: &[S]) -> Vec<usize>;

    /// 批量评估
    fn evaluate_batch(&mut self, states: &[S]) -> Vec<f64>;
}

/// 搜索算法接口
pub trait SearchAlgorithm<S: State, H: LLMHeuristic<S>> {
    fn search(&mut self, initial: S) -> Option<(Vec<S>, SearchStats)>;
}

/// 状态接口
pub trait State: Clone + Debug + PartialEq + Eq + Hash {
    fn is_goal(&self) -> bool;
    fn successors(&self) -> Vec<Self>;
    fn state_id(&self) -> String;
}
```

### 6.3 性能监控

```rust
#[derive(Debug, Clone, Default)]
pub struct SearchStats {
    pub nodes_expanded: usize,
    pub nodes_generated: usize,
    pub heuristic_evaluations: usize,
    pub cache_hits: usize,
    pub cache_misses: usize,
    pub search_time_ms: u64,
}

impl SearchStats {
    pub fn cache_hit_rate(&self) -> f64 {
        let total = self.cache_hits + self.cache_misses;
        if total == 0 { 0.0 } else { self.cache_hits as f64 / total as f64 }
    }

    pub fn avg_evaluations_per_node(&self) -> f64 {
        self.heuristic_evaluations as f64 / self.nodes_expanded.max(1) as f64
    }
}
```

### A.1 LLM-A* 核心实现分析

基于 `drafts/20260310_1200_llm_navigator_algo.rs` 的代码分析：

```rust
/// LLM-A*搜索器
pub struct LLMStarSearch<H: LLMHeuristic> {
    heuristic: H,
    config: AStarConfig,
    open_set: BinaryHeap<AStarNode>,
    closed_set: HashSet<StateId>,
    g_scores: HashMap<StateId, f64>,
    came_from: HashMap<StateId, StateId>,
    states: HashMap<StateId, StateContext>,
    stats: SearchStats,
}

impl<H: LLMHeuristic> LLMStarSearch<H> {
    pub fn search<F, G>(
        &mut self,
        initial: StateContext,
        goal_check: F,
        expand_fn: G,
    ) -> Option<(Vec<StateId>, SearchStats)>
    where
        F: Fn(&str) -> bool,
        G: Fn(&StateContext) -> Vec<String>,
    {
        // 1. 初始化
        let h0 = self.heuristic.evaluate(&initial);
        self.open_set.push(AStarNode::new(initial_id, 0.0, h0, None, 0));

        while let Some(current) = self.open_set.pop() {
            // 2. 目标检查
            if goal_check(&ctx.description) {
                return Some((self.reconstruct_path(current_id), self.stats.clone()));
            }

            // 3. 早停优化
            if let Some(threshold) = self.config.early_termination_threshold {
                if current.h_cost < threshold {
                    return Some((self.reconstruct_path(current_id), self.stats.clone()));
                }
            }

            // 4. 扩展邻居
            for action in actions {
                let h = self.heuristic.evaluate(&new_ctx);
                self.open_set.push(AStarNode::new(
                    new_id, tentative_g, h, Some(current_id), current.depth + 1,
                ));
            }
        }
    }
}
```

**关键优化点**:
1. **早停机制**: 当启发式值低于阈值时立即停止，避免不必要的搜索
2. **批处理评估**: `batch_evaluation_size`配置支持批量评估
3. **深度限制**: `max_depth`防止无限深度搜索
4. **迭代限制**: `max_iterations`防止无限循环

### A.2 Beam Search多样性惩罚实现

```rust
/// 计算多样性惩罚，避免beam中的节点过于相似
fn calculate_diversity_penalty(&self, ctx: &StateContext, candidates: &[BeamNode]) -> f64 {
    let mut penalty = 0.0;
    for candidate in candidates {
        if let Some(candidate_ctx) = self.states.get(&candidate.state_id) {
            let similarity = self.string_similarity(&ctx.description, &candidate_ctx.description);
            penalty += similarity * self.config.diversity_penalty;
        }
    }
    penalty
}

fn string_similarity(&self, s1: &str, s2: &str) -> f64 {
    // 简化的Jaccard相似度
    let words1: HashSet<_> = s1.split_whitespace().collect();
    let words2: HashSet<_> = s2.split_whitespace().collect();

    let intersection: HashSet<_> = words1.intersection(&words2).collect();
    let union: HashSet<_> = words1.union(&words2).collect();

    intersection.len() as f64 / union.len() as f64
}
```

**设计原理**:
- 使用Jaccard相似度衡量状态描述的相似性
- 相似度越高，惩罚越大
- 鼓励beam中的节点探索不同路径
- 防止所有束收敛到局部最优

### A.3 MCTS UCT公式实现

```rust
impl MCTSNode {
    pub fn uct_value(&self, parent_visits: usize, exploration_constant: f64) -> f64 {
        if self.visits == 0 {
            return f64::INFINITY;
        }

        let exploitation = self.total_reward / self.visits as f64;
        let exploration = exploration_constant *
            ((2.0 * (parent_visits as f64).ln()) / self.visits as f64).sqrt();

        exploitation + exploration
    }
}
```

**与传统MCTS的区别**:
1. **Simulation阶段**: 使用LLM启发式替代随机rollout
2. **Expansion阶段**: 使用启发式排序选择最佳未尝试动作
3. **价值函数**: 启发式值转换为奖励信号

pub struct PatternNavigator<H: LLMHeuristic> {

impl<H: LLMHeuristic> PatternNavigator<H> {
    /// LLM在受限的模式空间中选择
    pub fn navigate(&self, context: &str) -> &'static str {
        let mut best_pattern = self.available_patterns[0];
        let mut best_score = f64::INFINITY;

        for &pattern in &self.available_patterns {
            let state_ctx = StateContext {
                description: format!("Use {} pattern for {}", pattern, context),
                depth: 0,
                parent: None,
            };

            let score = self.heuristic.evaluate(&state_ctx);
            if score < best_score {
                best_score = score;
                best_pattern = pattern;
            }
        }

        best_pattern
    }
}

## 补充发现：BFS-Prover (2025年2月)

**论文**: "BFS-Prover: Scalable Best-First Tree Search for LLM-based Automatic Theorem Proving"
- 作者: Ran Xin, Chenguang Xi, Jie Yang, Feng Chen等 (ByteDance/Stanford)
- arXiv: 2502.03438

**核心发现**:

BFS-Prover挑战了"复杂搜索方法(MCTS)必然更好"的传统认知：

1. **长度归一化BFS**:
   ```
   score(s_L) = sum(log p(a_t|s_t)) / L^α
   ```
   其中α∈[0,1]是可调参数，解决BFS对长路径的偏见

2. **专家迭代+自过滤**:
   - 过滤掉可通过beam search解决的简单问题
   - 专注于更难的问题
   - 每轮迭代提升策略LLM

3. **DPO优化**:
   - 利用Lean编译器错误反馈
   - 正样本: 证明路径上的tactic
   - 负样本: 导致编译器错误的tactic

**实验结果**:
- MiniF2F测试集: **71.31分** (SOTA)
- 无需critic model或MCTS
- 证明长度分布随迭代增加(平均10.2→16.7 tactics)

**对LLM导航器的启示**:
1. BFS可能比MCTS更适合某些结构化搜索任务
2. 长度归一化是关键优化
3. 编译器/验证器反馈可作为DPO的偏好信号
4. 专家迭代+数据过滤可持续提升性能
