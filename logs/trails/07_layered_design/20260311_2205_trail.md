## Step 1: Web Research (8-10分钟)

### 1.1 编译器分层架构（LLVM IR、MLIR）

**关键发现**:
- MLIR采用多层级方言(Dialect)架构，支持渐进式Lowering
- 2024年Pattern-Based IR Rewriting框架增强，支持显式转换管道调度
- 两阶段Lowering流程：Conversion Stage → Translation Stage

**核心贡献**:
1. **LLVM/MLIR**: 通过Dialect系统实现多层级IR，每层有明确的抽象级别和优化职责
2. **CompCert验证编译器**: 机器检查证明实现语义保持，使用简化类型系统确保每层well-formedness
3. **ANF转换**: 保持依赖类型"直到外延等价"，处理宇宙层级(Type₀:Type₁:Type₂)

**技术方案差异**:
| 系统 | 抽象级别 | 类型安全 | 验证方法 |
|------|----------|----------|----------|
| MLIR | 多级Dialect | 动态检查 | 测试验证 |
| CompCert | 单级IR | 静态证明 | 形式化验证 |
| CakeML | 多级 | 静态 | 定理证明 |

### 1.2 类型系统层次设计

**关键发现**:
- Universe Hierarchy: 预测性层级，抽象类型生活在更高宇宙
- Type-Preserving Compilation: 证明每个编译pass保持类型和语义
- 语义保持技术：Denotational semantics、Logical relations、Bisimulations

### 1.3 状态机分层实现模式（Rust）

**关键发现**:
- **Type-State Pattern**: 编译时状态转换安全，零运行时开销
- **分层状态机**: 外层Enum包装 + 内层泛型状态机
- **Trait-Based异步状态机**: 支持async/await的事件驱动架构

## Step 2: 提出假设（3-5分钟）

### 技术假设
**分层转换如何保持语义不变性？**
- 假设：通过在每个转换层定义形式化语义映射函数，可以证明层间转换的语义保持性
- 关键机制：使用逻辑关系(Logical Relations)或双模拟(Bisimulation)证明行为等价
- 验证方法：实现四层转换并证明关键不变量

### 实现假设
**Rust中如何表达层间转换？**
- 假设：使用Rust的trait系统可以类型安全地表达层间转换
- 设计：每层实现`Into<NextLayer>` trait，利用类型状态模式确保有效转换
- 预期：编译器在编译期拒绝无效转换路径

### 性能假设
**分层设计的运行时开销？**
- 假设：使用泛型和零大小类型(ZST)，分层抽象在运行时零开销
- 预期：所有层间转换在编译期完成，运行时代码与手写单层代码等价
- 验证：对比分层实现与扁平实现的汇编输出

### 适用性假设
**适用于什么场景？**
- 假设：适用于需要多阶段转换的编译器、协议解析器、状态机系统
- 不适用：简单的一次性转换，分层带来不必要的复杂度
- 关键指标：当转换逻辑复杂且需要可验证性时，分层设计ROI为正

## Step 3: 验证（10-12分钟）

### 代码实现

创建了完整的四层转换实现：`drafts/20260311_2205_layered_design.rs`

**四层架构**:

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: Syntax Layer (抽象语法树)                           │
│ ├─ SyntaxNode: 语法节点                                       │
│ ├─ SyntaxKind: 节点类型枚举                                   │
│ └─ parse(): 源代码解析                                        │
├─────────────────────────────────────────────────────────────┤
│ Layer 2: Semantic Layer (语义表示)                           │
│ ├─ SemanticExpr: 类型化表达式                                 │
│ ├─ SymbolTable: 符号表管理                                    │
│ └─ lower_syntax_to_semantic(): 语法→语义转换                  │
├─────────────────────────────────────────────────────────────┤
│ Layer 3: Pattern Layer (模式表示)                            │
│ ├─ Pattern: 计算模式枚举 (Sequential/Parallel/Map/Reduce)     │
│ ├─ PatternProperties: 模式属性分析                            │
│ └─ lower_semantic_to_pattern(): 语义→模式转换                 │
├─────────────────────────────────────────────────────────────┤
│ Layer 4: Domain Layer (领域特定实现)                         │
│ ├─ TargetDomain: 目标领域枚举 (Cpu/Gpu/Fpga/Distributed)      │
│ ├─ DomainCode: 领域特定代码                                   │
│ └─ lower_to_domain(): 模式→领域代码生成                       │
└─────────────────────────────────────────────────────────────┘
```

**关键设计决策**:

1. **Type-State模式**: 使用`PhantomData<Context>`标记每层上下文，确保类型安全
2. **TryFrom trait**: 层间转换使用标准`TryFrom` trait，统一错误处理
3. **语义保持证明**: `SemanticPreservationProof<From, To>`标记编译期证明

**验证的假设**:
1. ✅ 技术假设：分层转换可以通过类型系统保持语义不变性

**关键洞察**:
1. **分层边界清晰**: 每层有明确职责，通过类型系统强制执行

3. **目标无关优化**: Pattern层可以在 lowering 前进行目标无关优化
