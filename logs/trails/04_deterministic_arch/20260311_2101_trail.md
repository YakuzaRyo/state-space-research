#### 三层Hook系统

| Level | Hook Script | Scope | Function |
|-------|-------------|-------|----------|
| **L1: Intra-Task** | `iteration-limit-stop.sh` | Single agent | Prevents endless spinning (max 10 iterations) |
| **L2: Inter-Phase** | `feedback-loop-stop.sh` | Implementation→Review→Test cycle | Blocks exit until independent validation passes |
| **L3: Orchestrator** | Skill logic | 16-phase workflow | Re-invokes phases if macro-goals missed |

### Step 2: 提出假设 (3-5分钟)

**H1: Thin Agent的边界清晰性假设**
Thin Agent (<150行)与Fat Platform的边界应基于"状态所有权"而非"功能复杂度"。Thin Agent是纯粹的状态转换函数 `f(state, input) -> (new_state, output)`，所有副作用通过消息委托给Platform。

**H2: Gateway模式的确定性路由假设**
Gateway通过"意图匹配"而非"硬编码路径"实现技能路由，其确定性来自于规则匹配引擎在LLM上下文外执行。

**H3: 三层Hook系统的层级防御假设**
- L1 (Intra-Task): 防止无限循环，基于迭代计数
- L2 (Inter-Phase): 强制反馈循环，基于状态验证
- L3 (Orchestrator): 工作流编排，基于16阶段状态机

**H4: 权限分离的互斥性假设**
Coordinator (can_task=true, can_edit=false) 与 Executor (can_task=false, can_edit=true) 的权限互斥是安全的关键不变量。

**H5: 状态空间模型的适用性假设**
控制理论中的状态空间方程 `x(k+1) = Ax(k) + Bu(k)` 可用于建模Agent状态转换，其中Platform维护状态向量x，Agent提供控制输入u。

**关键发现:**

1. **Thin Agent的精确边界**: 不是简单的代码行数限制，而是"状态所有权"的转移。Agent是 `f(state, input) -> output` 的纯函数，所有状态由Platform维护。

2. **Gateway的确定性保证**: 意图匹配使用简单的关键词规则(非LLM)，确保路由决策在LLM上下文外执行，避免非确定性。

3. **三层Hook的层级关系**:
   - L1保护单个Agent不陷入无限循环
   - L2保护跨阶段反馈循环完整性
   - L3保护工作流级别的正确性

4. **权限分离的核心价值**: "An agent cannot be both coordinator and executor" 是防止自我修改导致的不安全状态的关键不变量。

5. **状态空间模型的适用性**: 控制理论中的 `x(k+1) = Ax(k) + Bu(k)` 可以有效建模Agent行为，Platform维护状态向量，Agent提供控制输入。
