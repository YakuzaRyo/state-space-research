### 4. 输出结果

#### 核心精化法则 (Morgan's Calculus)

| 法则 | 形式化表述 |
|------|-----------|
| Skip | 若 pre => post，则 w:[pre, post] ⊑ skip |
| Assignment | 若 pre => post[x:=E]，则 w:[pre, post] ⊑ x:=E |
| Sequential | w:[pre, post] ⊑ w:[pre, mid]; w:[mid, post] |
| Strengthen Post | 若 post' => post，则 w:[pre, post] ⊑ w:[pre, post'] |
| Weaken Pre | 若 pre => pre'，则 w:[pre, post] ⊑ w:[pre', post] |

#### Weakest Precondition演算

```
wp(skip, Q) = Q
wp(x:=E, Q) = Q[x:=E]
wp(S1; S2, Q) = wp(S1, wp(S2, Q))
wp(if B then S1 else S2, Q) = (B => wp(S1,Q)) ∧ (¬B => wp(S2,Q))
```

#### 验证条件生成

```
VC: pre => wp(implementation, post)
```

### 5. 调整方向

#### 关键洞见

1. **精化演算提供形式化"脚手架"**: LLM在规约约束下生成，而非自由生成
2. **VC生成是桥梁**: 将形式化规约转化为可自动验证的条件
3. **反馈循环关键**: 反例驱动的修正使LLM逐步逼近正确实现
4. **分层精化降低复杂度**: 高层规约逐步分解，LLM每次只需解决子问题

#### 与LLM集成的工作流程

```
1. 高层规约 → 精化引擎分解
   ↓
2. 生成中间规约和约束
   ↓
3. LLM接收 (规约, 约束, 上下文)
   ↓
4. LLM生成代码候选
   ↓
5. VC生成器产生验证义务
   ↓
6. SMT求解器验证 (Z3/CVC4)
   ↓
7. 若失败：反例反馈给LLM修正
   ↓
8. 若成功：代码通过验证
```
