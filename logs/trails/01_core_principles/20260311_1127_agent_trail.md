### 关键发现

#### 发现1: Miri (POPL 2026) - 实用化UB检测
- **来源**: https://research.ralfj.de/papers/2026-popl-miri.pdf
- **核心贡献**: 首个可检测所有确定性Rust程序中未定义行为的工具
- **关键数据**: 测试10万+库，执行>70%测试，发现数十个真实bug
- **与本研究关联**: 为L0-L2层提供动态验证补充
- **意外发现**: Miri已集成到Rust标准库CI，成为基础设施

#### 发现2: THRUST (PLDI 2025) - 自动化精化类型
- **来源**: https://www.riec.tohoku.ac.jp/~unno/papers/pldi2025.pdf
- **核心贡献**: 基于预言变量的全自动依赖精炼类型系统
- **技术差异**: 相比Verus需要手写规约，THRUST全自动推断
- **与本研究关联**: L4层自动化方向，降低形式验证门槛

#### 发现3: CAPsLock (CCS 2025) - 硬件辅助能力安全
- **来源**: https://www.comp.nus.edu.sg/~prateeks/papers/CapsLock.pdf
- **核心贡献**: 在机器码级别强制执行Rust所有权原则
- **关键数据**: 99.7%兼容100个最流行的Rust crate，发现8个未知bug
- **技术差异**: 软件cap-std vs 硬件CAPsLock的防御深度差异
- **与本研究关联**: L5层硬件化方向，软硬件协同安全

#### 发现4: AutoVerus (2025) - LLM辅助形式验证
- **来源**: http://jaylorch.us.s3.amazonaws.com/publications/autoverus.pdf
- **核心贡献**: 使用GPT-4o自动生成Rust代码证明
- **关键数据**: 150个非平凡证明任务成功率>90%
- **意外发现**: LLM不仅可以生成代码，还可以生成形式验证证明
- **与本研究关联**: 假设9（LLM+类型约束）的间接支持证据

#### 发现5: Rust Types Team Roadmap (2024-2027)
- **来源**: https://blog.rust-lang.org/2024/06/26/types-team-update/
- **核心贡献**: 官方类型系统发展路线图
- **关键里程碑**:
  - 2025: 下一代trait求解器默认启用、TAIT稳定化
  - 2027: 协归纳支持、所有已知类型系统不健全性修复
- **与本研究关联**: L1-L3层的语言级改进

#### 发现6: A Hybrid Approach to Semi-automated Rust Verification (PLDI 2025)
- **来源**: https://pldi25.sigplan.org/details/pldi-2025-papers/40/A-Hybrid-Approach-to-Semi-automated-Rust-Verification
- **核心贡献**: 半自动化验证的混合方法
- **与本研究关联**: L4层实用化路径

## Step 2: 提出假设 (计划: 3-5分钟, 实际: 约5分钟)

### 假设列表

#### H1: 技术假设
**内容**: 六层渐进式边界模型可实现零成本错误预防
**置信度**: 高
**理由**:
- PhantomData是ZST（零大小类型）
- const generics在编译期计算，无运行时开销
- Rust单态化后类型擦除，状态检查完全在编译期完成
**证伪条件**: 发现任何一层有非零运行时开销的反例

#### H2: 实现假设
**内容**: Typestate + Capability 组合可消除90%以上的运行时状态错误
**置信度**: 高
**理由**:
- Typestate将状态错误转为编译期错误
- Capability将权限错误转为编译期错误
- 两者组合覆盖状态和权限两类常见错误
**边界条件**: 不适用于动态状态（如用户输入决定的状态）和分布式场景（序列化后类型丢失）
**证伪条件**: 在业务状态机中发现无法被编译期捕获的状态错误

#### H3: 适用性假设（新）
**内容**: 编译期约束对LLM代码生成质量有正向影响
**置信度**: 中
**理由**:
- AutoVerus证明LLM可以理解和生成形式验证代码
- 类型约束缩小搜索空间，可能提高生成正确性
- 但可能限制LLM的"创造性"
**证伪条件**: 对照实验显示约束生成质量低于自由生成

### 假设与2025-2026研究的关联

| 假设 | 相关研究 | 支持证据 |
|------|---------|---------|
| H1 | Miri测试10万+库 | 零成本抽象在工业规模验证 |
| H2 | cap-std, CAPsLock | 能力安全在真实项目中的应用 |
| H3 | AutoVerus | LLM可以理解和生成约束代码 |

```rust
pub struct BoundedU32<const MIN: u32, const MAX: u32>(u32);
```

**L1验证代码**:
```rust

pub struct UserId(u64);

```
**结果**: ✅ 通过 - Newtype零成本，编译期类型区分

**L3验证代码**:
```rust
pub struct TypedFile<S> {
    path: String,
    _state: PhantomData<S>,  // ZST
}
impl TypedFile<FileClosed> {
    pub fn open(self) -> TypedFile<FileOpen> { ... }
}
```
**结果**: ✅ 通过 - PhantomData是ZST，单态化后无运行时开销

#### 验证H2: Typestate + Capability组合

**Typestate验证**:
```rust
let file = TypedFile::<FileClosed>::new("test.txt");
let file = file.open();
let mut file = file.start_writing();
file.write("Hello");
// file.read() // 编译错误！写模式不能读
```
**结果**: ✅ 通过 - 无效状态转换在编译期被拒绝

**Capability验证**:
```rust

let resource = resource.grant_read();
// resource.write("new") // 编译错误！没有写权限
```
**结果**: ✅ 通过 - 无权限操作在编译期被拒绝

**组合验证 - 权限状态机**:

```rust
draft.edit("updated");  // 草稿状态可编辑

// review.edit("changed") // 编译错误！审核状态只读
```
**结果**: ✅ 通过 - 状态和权限联合约束

### 边界条件确认

1. **序列化边界**: 类型状态在序列化后丢失，需要运行时schema验证补充
2. **动态状态**: 用户输入决定的状态无法完全用Typestate表达
3. **复杂类型签名**: 嵌套状态机可能产生复杂的类型签名，影响可读性
