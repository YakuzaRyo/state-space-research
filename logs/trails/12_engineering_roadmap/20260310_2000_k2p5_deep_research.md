### 2.2 关键依赖关系详解

#### 依赖1: 01_core_principles → 所有其他方向

**依赖类型**: 理论基础依赖
**关键传递**:
- 六层渐进式边界模型 (L0-L5) 为所有实现提供安全层级框架
- 硬性边界理念指导各层的设计决策
- "让错误在设计上不可能发生" 作为核心设计原则

**代码体现**:
```rust
// L0: Const Generics 编译期常量约束
pub struct BoundedU32<const MIN: u32, const MAX: u32>(u32);

// L3: Typestate 编译期状态机
pub struct Connection<S: ConnectionState> { ... }

// L5: Capability 权限系统
pub struct PermissionVector<ReadMode, WriteMode> { ... }
```

#### 依赖2: 07_layered_design ↔ 08_llm_as_navigator

**依赖类型**: 双向协作依赖
**协作机制**:
- 分层架构提供状态空间的分层表示
- LLM导航器在Pattern层选择优化策略
- 导航结果反馈到Domain层执行

**接口契约**:
```
L7 Layered Design                    L8 LLM Navigator
─────────────────                    ─────────────────
Syntax Layer (Token)        ───────►  State定义
Semantic Layer (AST)        ◄──────►  Heuristic评估
Pattern Layer (优化)         ◄──────   策略选择
Domain Layer (执行)         ◄──────   行动执行
```

#### 依赖3: 03_structured_generation + 05_type_constraints → 12_engineering_roadmap

**依赖类型**: 能力集成依赖
**集成点**:
- XGrammar引擎提供Token级别的结构化约束
- 类型约束生成确保代码类型安全
- 两者结合实现"类型驱动的结构化生成"

**工程架构中的位置**:
```rust
// Layer 3: LLM集成层 (12_engineering_roadmap.rs)
pub trait LlmClient: Send + Sync {
    // 结构化输出由03_structured_generation提供支持
    async fn generate_structured<T: DeserializeOwned>(
        &self,
        prompt: &str,
        schema: &serde_json::Value,
    ) -> Result<T, LlmError>;
}
```

#### 依赖4: 09_rust_type_system → 06_formal_verification

**依赖类型**: 技术栈协同依赖
**协同效应**:
- Rust所有权系统简化形式验证 (无需分离逻辑)
- Typestate模式为验证提供编译期基础
- Kani/Verus工具链与Rust类型系统深度集成

**验证工具选择矩阵**:
| 场景 | 工具 | 与Typestate的协同 |
|------|------|------------------|
| unsafe代码边界 | Kani | 验证unsafe块的安全性 |
| 并发协议 | Verus | 验证Typestate状态转换 |
| 算法正确性 | Creusot | 验证纯函数语义 |

## 三、工程实现策略分析

### 3.1 六层工程架构 (来自12_engineering_roadmap.rs)

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 6: MCP Integration (MCP集成层)                         │
│ - McpGateway: 状态空间到MCP的适配器                          │
│ - StateMcpAdapter trait: 双向转换接口                        │
├─────────────────────────────────────────────────────────────┤
│ Layer 5: Observability (可观测性层)                          │
│ - TraceContext: 分布式追踪                                   │
│ - MetricsCollector: Counter/Histogram/Gauge                  │
├─────────────────────────────────────────────────────────────┤
│ Layer 4: Caching (缓存层)                                    │
│ - SemanticCache: 余弦相似度匹配                              │
│ - LRU淘汰策略                                               │
├─────────────────────────────────────────────────────────────┤
│ Layer 3: LLM Integration (LLM集成层)                         │
│ - CircuitBreaker: 故障隔离 (Closed/Open/HalfOpen)            │

├─────────────────────────────────────────────────────────────┤
│ Layer 2: Actor System (Actor系统层)                          │

├─────────────────────────────────────────────────────────────┤
│ Layer 1: Core State Machine (状态机核心层)                   │
│ - StateMachine: 状态转换引擎                                 │
│ - Transition: 转换规则 + 条件 + 动作                         │
│ - History: 状态回滚机制                                      │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 实现优先级与依赖关系

**验证结果**: H1 (六层架构优先级) **部分验证**

| 层级 | 组件 | 优先级 | 依赖 | 可并行性 |
|------|------|--------|------|----------|
| L1 | StateMachine | P0 | 无 | 串行 |
| L2 | Actor System | P0 | L1 | 串行 |
| L3 | LLM + CircuitBreaker | P0 | L2 | 串行 |
| L4 | SemanticCache | P1 | L3 | 可与L3并行 |
| L5 | Observability | P1 | L3 | 可与L4并行 |
| L6 | MCP Gateway | P2 | L1-L5 | 最后实现 |

**关键发现**:
- L1和L2必须严格顺序实现 (状态机是基础，Actor依赖状态机)
- L3和L4可以并行开发 (缓存层不依赖LLM集成细节)
- L6 (MCP) 需要等待所有下层稳定后再实现

### 3.3 技术栈选择矩阵

| 层级 | 推荐技术 | 备选方案 | 选择理由 |
|------|----------|----------|----------|
| 语言 | Rust | TypeScript | 类型安全 + 性能 |
| 运行时 | Tokio | async-std | 生态成熟 |
| 序列化 | serde + schemars | prost | JSON生态 |
| LLM集成 | 原生HTTP + XGrammar | LiteLLM | 结构化生成 |
| 缓存 | Redis | 内存LRU | 语义缓存 |
| 向量存储 | Qdrant | pgvector | 性能 |
| 可观测性 | tracing + OTel | Logfire | 标准化 |

## 四、关键工程决策与权衡

### 4.1 类型系统复杂度 vs 易用性

**决策**: 编译时类型安全 + 运行时灵活验证

```rust
// 编译时类型安全: 防止非法状态转换
impl StateMachine<Created> {
    pub fn initialize(self) -> StateMachine<Initialized> { ... }
}

// 运行时灵活验证: 支持动态规则
pub struct ValidationRule {
    condition: fn(&State, &TransitionTrigger) -> bool,
}
```

**权衡**: 开发时严格，运行时灵活

### 4.2 性能优化策略

**延迟优化**:
- 语义缓存: 相似查询直接返回 (30%成本降低，15x响应加速)
- Prompt缓存: KV Cache复用 (90%成本降低，85%延迟降低)
- 预编译Grammar: XGrammar模式

**吞吐量优化**:
- Actor并发模型
- 连接池管理
- 批处理请求

### 4.3 与现有工具链集成

**MCP (Model Context Protocol)**:
- Anthropic主导的行业标准
- 2025年已被OpenAI、Google采纳
- 需要Gateway适配器层实现与状态空间的结合

**结构化生成**:
- XGrammar/Outlines集成
- JSON Schema约束输出
- 消除LLM输出不确定性

## 九、研究总结

### 9.1 核心洞察

1. **工程化路径确认**: 六层架构是可行的，但需要严格的实现顺序 (L1/L2串行，L3/L4可并行)

2. **模块依赖清晰**: 01_core_principles作为理论基础，07_layered_design作为架构骨架，其他方向在此基础上扩展

3. **MCP集成策略**: 需要适配器层，但不是简单的包装，需要重新设计边界 (StateMcpAdapter trait)

4. **确定性运行时可行**: Praetorian Gateway模式在Rust中可以有效实现 (CircuitBreaker + StateMachine)

5. **生产现实严峻**: 根据Cleanlab研究，当前Agent生产化仍处于早期阶段，可靠性是最大挑战
